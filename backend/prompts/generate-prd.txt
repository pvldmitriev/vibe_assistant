ВЛИЯНИЕ ЦЕЛИ НА PRD:

{{#if goal == "Обучение и практика"}}
ПОДХОД ДЛЯ ОБУЧЕНИЯ:
- Технологии: выбирай простые и популярные (Node.js, React, простой стек)
- Функционал: только core MVP, без усложнений
- Качество: "пусть работает", не overengineering
- Тесты: НЕ требуются
- Документация: минимальная (README с quick start)
- Фокус: на понимании основ и быстром результате
{{/if}}

{{#if goal == "Использовать самому"}}
ПОДХОД ДЛЯ ЛИЧНОГО ИСПОЛЬЗОВАНИЯ:
- Технологии: удобные для поддержки одним человеком
- Функционал: то что реально нужно пользователю
- Качество: стабильный рабочий код
- Тесты: только для критичных частей (опционально)
- Документация: для себя (как запустить и настроить)
- Фокус: на практичности и удобстве
{{/if}}

{{#if goal == "Для пользователей"}}
ПОДХОД ДЛЯ ПРОДАКШН:
- Технологии: production-ready, масштабируемые, надежные
- Функционал: MVP + полная обработка ошибок + валидация
- Качество: best practices, безопасность, производительность
- Тесты: ОБЯЗАТЕЛЬНЫ (unit + integration для всех ключевых функций)
- Документация: API docs, troubleshooting, FAQ
- Фокус: на надежности и качестве
{{/if}}

{{#if goal == "Портфолио"}}
ПОДХОД ДЛЯ ПОРТФОЛИО:
- Технологии: современные, впечатляющие, актуальные
- Функционал: демонстрирует технические навыки
- Качество: идеальный код, паттерны, архитектура, типизация
- Тесты: ОБЯЗАТЕЛЬНЫ (высокое покрытие, unit + integration + e2e)
- Документация: подробный README, архитектура, примеры кода
- Фокус: код должен впечатлять рекрутеров
{{/if}}

ТЕХНИЧЕСКИЕ РЕШЕНИЯ (ты принимаешь сам по best practices):

{{#if category == "BOT"}}
ДЛЯ TELEGRAM БОТА:
- Платформа: всегда Telegram
- Библиотека: telegraf (Node.js) или аналог
- Старт: команда /start
- БД: определи нужна ли на основе функционала (нужно ли хранить данные пользователей?)
- Деплой: зависит от цели (обучение = локально, продакшн = облако)
{{/if}}

{{#if category == "MOBILE_APP"}}
ДЛЯ МОБИЛЬНОГО ПРИЛОЖЕНИЯ:
- Платформа: выбери iOS или Android (что логичнее для целевой аудитории)
- НЕ делай кросс-платформу (только если цель = портфолио)
- Технологии: Swift/SwiftUI для iOS или Kotlin для Android
- Офлайн: реши сам нужен ли на основе use case
- Деплой: TestFlight (iOS) или internal testing (Android)
{{/if}}

{{#if category == "WEB_APP"}}
ДЛЯ ВЕБ-ПРИЛОЖЕНИЯ:
- Стек: выбери оптимальный (Next.js/React для современного, или что лучше подходит)
- Auth: определи нужна ли на основе описания продукта
- Страницы: продумай минимальный набор для MVP
- Интеграции: определи какие внешние API нужны
- Деплой: зависит от цели (обучение = локально, продакшн/портфолио = Vercel)
{{/if}}

СТРУКТУРА PRD (обязательные разделы):

# [Название продукта]

## Описание
Краткое описание продукта (2-3 предложения)

## Целевая аудитория
Для кого это? Какие у них потребности?

## Проблема
Какую конкретную проблему решаем?

## Решение
Как наш продукт решает эту проблему?

## Основной функционал (MVP)
Список конкретных функций (учитывай цель проекта):
- Функция 1
- Функция 2
- ...

{{#if goal == "Для пользователей" || goal == "Портфолио"}}
## Обработка ошибок
- Валидация входных данных
- Понятные сообщения об ошибках
- Graceful degradation
{{/if}}

## Технологический стек
Обоснованный выбор технологий с пояснением "почему":
- Frontend: [технология] - потому что [причина]
- Backend: [технология] - потому что [причина]
- База данных: [технология или "не требуется"] - потому что [причина]
- Деплой: [где] - потому что [причина]

## User Flow
Пошаговое описание как пользователь взаимодействует с продуктом:
1. Пользователь...
2. Система...
3. ...

{{#if goal == "Для пользователей" || goal == "Портфолио"}}
## Требования к качеству
- Безопасность: [что важно]
- Производительность: [требования]
- Тестирование: [что тестируем]
{{/if}}

## Критерии успеха MVP
Как понять что MVP готов:
- Критерий 1
- Критерий 2
- ...

{{#if goal == "Портфолио"}}
## Особенности для портфолио
- Архитектурные решения достойные внимания
- Демо-данные для показа
- Ссылка на live demo
{{/if}}
